Not an admin thing — it's a browser-sandbox thing, and it's deliberate. Mouse DPI/sensitivity lives at the hardware/driver/OS layer. A webpage (and a content script is just code running in webpages) receives pointer events; it has zero API access to *change* how the pointer behaves. Even with full admin rights, JavaScript in a page can't touch it — you'd need a native companion app outside the browser (Windows exposes pointer speed via `SystemParametersInfo`, and true DPI needs the mouse vendor's software). Browsers lock this down on purpose: a webpage that could change your mouse sensitivity is a webpage that can make your mouse unusable.

There IS one in-browser trick worth knowing about: the **Pointer Lock API** — a page captures the mouse, hides your real cursor, and renders a *virtual* cursor it can move with any speed multiplier it wants. That's how browser FPS games do it. But it requires a click to engage, throws up a "mouse is captured" banner, Esc breaks it, and your real cursor stops interacting with the page. For Neckbeard that's a deal-breaker on identity grounds, not just friction: the whole joke is that he's chasing your *actual* cursor while you're *actually* browsing. The moment we swap in a virtual cursor, it's a game overlaid on a webpage, not a haunted webpage.
